## 🔴🔴 做T优先原则（2026-07-17确立）

**用户明确偏好：亏损时做T降成本，不是割肉止损。**

用户原话："我要做t你不理解吗？你现在让我割肉？"、"你总说等企稳 等稳定了还有抄底什么事？"

| 情境 | ❌ 旧做法 | ✅ 新做法 |
|------|----------|----------|
| 持仓亏损+大盘暴跌 | 建议止损 | 建议做T：低位补仓→反弹卖旧份额 |
| RSI超卖+近10日低点 | 说"风险大不补" | 主动算补仓价和T目标 |
| T目标到达 | 等用户问 | **立即推企微** |

**⚠️ 除非用户明确说"割肉"，否则默认走做T路径。**

### 做T核心流程（补仓→反弹→卖旧份额）

```
今天低位补仓 → 补的份额锁仓(T+1)
↓
设T目标 = 补仓价 + 2~5%
↓
价格反弹到T目标 → cronjob推企微
↓
卖出旧持仓中对应数量 → 总份额不变，成本下降
```

### T目标推送格式（必须写清楚！）

用户原话："是让我多少卖啊 你也没写清楚啊" — 推送必须含具体股数和价格。

```
🎯 【股票名 做T卖出指令】
   卖出 200股 @ 3.03
   （补200@2.974  T赚+11）
```

**⚠️ 推送时机**：计算好T目标后→立刻推企微；价格到达时→cronjob自动推。不要等用户问"推送啊你倒是"。

## ⚠️ T+1与周末锁仓风险（2026-06-26教训）

用户指出："还有100股跌停价挂单啊"、"你不该考虑下T+1吗"、"明天是交易日吗"。

### 核心规则
- **T+1**: 当天买入的股票，次日才能卖出
- **周五买入**: 锁仓2天（周六、周日），下周一才能操作
- **节假日前买入**: 锁仓更久

### 分析时必须区分
```
持仓 = 可卖部分（昨日及之前买入） + 锁仓部分（今日买入）

例如：
- 原持仓200股（可卖）
- 今日新买100股（锁仓到下周一）
- 止损时只能卖200股，100股被锁住
```

### 操作建议中的应用
- **止损建议**: 只能针对可卖部分，锁仓部分无法操作
- **加仓建议**: 周五买入需考虑锁仓风险（周末利空无法卖出）
- **风险敞口**: 锁仓部分=不可控风险，周末/假期前谨慎加仓

### 🔴 教训8：减仓后必须摊薄成本（2026-07-15血泪教训）

亚威1200@10.99，冲高+5%卖600@11.32后，我把剩余600股成本写成10.99。用户立即纠正："卖出后成本会变得把"。

**✅ 正确：减仓盈利必须摊薄剩余成本**
```python
# 减仓后摊薄成本
new_cost = (old_shares * old_cost - sold_shares * sell_price) / remain_shares
# 亚威: (1200*10.99 - 600*11.32) / 600 = 10.66
```

**同步更新**：
- 成本变了 → 止损/止盈也重算（sl = cost*0.92, tp = cost*1.15）
- 止损止盈从基于买入价变为基于摊薄成本，价格全部下移
- 减仓≠只减股数，所有参数联动更新

**❌ 错误**：减仓后成本不变 → 止损止盈偏宽 → 用户实际风险被低估

### 🔴 教训9：减仓+清仓时TOTAL_CAPITAL双重计数陷阱（2026-07-15）

摊薄成本后，清仓剩余仓位时利润被重复计算：

```python
# 原始买入: 1200@10.99 = 13,188
# 第一笔卖: 600@11.32, 盈利+198 → TOTAL_CAPITAL: 72874→73072, cost摊薄→10.66
# 第二笔卖: 600@11.14, 系统算(11.14-10.66)*600=288 → 但这288包含了第一笔的198！
# ❌ 错误: TOTAL_CAPITAL = 73072 + 288 = 73360（多加了198）
# ✅ 正确: 从原始基础算总盈利 = (600*11.32+600*11.14)-1200*10.99 = 288
#           TOTAL_CAPITAL = 72874 + 288 = 73162
```

**规则**：清仓（最后一批卖出）时，TOTAL_CAPITAL 不要用摊薄成本算增量，必须回到**原始买入总成本**计算总盈亏，直接从最后一次减仓前的CAPITAL加总盈亏。

### 🔴 教训7：用户要实用工具，不要通用系统（2026-06-23）

**事件回顾：**
- 用户让我找GitHub上的股票分析系统
- 我找到了aiagents-stock（1500+ stars，多AI智能体分析系统）
- 部署后用户说"感觉这个系统不是很好用"
- 最终删除了这个系统，回到改进现有的监控脚本

**核心原则：**
1. **用户已有特定工具时，优先改进现有工具** — 不要引入通用系统
2. **3万本金的小资金需要具体建议** — 不是通用的技术分析报告
3. **持仓天数、仓位管理比AI分析更实用** — 用户需要的是执行纪律
4. **用户说"没啥用"时，立即调整方向** — 不要坚持推广

**正确做法：**
```
用户说"这个系统不好用" → 立即删除，改进现有工具
用户要"结合我的持仓" → 加入持仓天数提醒、仓位管理
用户要"具体怎么操作" → 给出价格、数量、时间
```

**错误做法：**
```
❌ 坚持推广通用分析系统
❌ 只给Web链接让用户自己看
❌ 输出泛泛的技术分析报告
```
