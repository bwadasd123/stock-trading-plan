### 🔴 做T分析必须用实时价格（2026-07-21教训）

**用户说"价格不对重新分析"——我用的是几分钟前的旧价格。**

做T方案涉及精确的接回价和卖出价，差0.05元就决定是否成交。
**规则**：发做T方案前，必须重新拉一次实时行情（push2delay API），贴最新价格后再算。不要用本会话中之前缓存的价格。

### 做T模式触发词
- "做T"、"补仓做T"、"补仓降成本"
- "能不能补仓"（结合上下文判断是做T还是加仓）

### 做T模式下的回答规范
- ✅ 给出补仓价、目标卖出价、T收益测算
- ✅ 设置T目标价监控，价格到了自动推企微
- ✅ 区分锁仓份额（今天买的）和可卖份额（昨天买的）
- ❌ 禁止说"止损"、"割肉"、"触发止损必须出"
- ❌ 禁止说"等企稳" — 用户原话："等稳定了还有抄底什么事？"

### 做T操作框架
```
今天低位补仓 → 下周一/二反弹 → 卖出等额旧份额 → 降成本
```
- 基金做T：按净值成交，先进先出，赎最早（最高成本）那批
- 股票做T：今天买锁T+1，卖旧份额（昨天买的今日可卖）
- 周五买入锁仓到周一，周末风险需提醒

### 做T目标设置后必须验证广播（2026-07-20教训）

设置t_targets/target_buy后，必须确认整点播报会显示进度距离。流程：
1. 更新 `state['t_targets']` 或 `price_monitor.py` 的 `target_buy`
2. ⚠️ 检查整点播报代码是否含做T目标距离显示（`price_monitor.py` 中 `current_hour != state.get("last_hourly")` 块）
3. 如果不含 → 用 patch 在 `format_stock + "\\n"` 之后追加距离行
4. 用户期待每小时看到"🎯 做T卖出 27.74 ⏳距-2.9%"而不是等触发

格式化规范：
```
🎯 做T卖出 {target} {✅/⏳}距{±x.x}%  {shares}股
🔻 补仓目标 {target} {🔔/⏳}距{±x.x}%  {shares}股
```
- ✅ = 已到达（dist≥0），🔔 = 补仓价已触发（dist≤0），⏳ = 等待中

### 不能做T的情况
- 大盘任一指數 ≤ -3% → 暂停，但只说不建议，不禁止（用户可能坚持）
- 已经持有3只（铁律④）且做T会再补一只 → 选优先做T的标的

## 🔴 分析前必须确认日期和时间（2026-07-20教训）

**错误**：用户问"现在怎么办"，我说"下周反弹"。但当时是周一，用户纠正："下周？你好好的检查一下你的逻辑"。

**✅ 规则**：
1. **给时间建议前先确认今天是周几**：`datetime.now().strftime('%A')`。周一不要说"下周"，今天就是本周一，本周还有4个交易日。
2. **每次给操作建议前必须拉最新实盘数据**，不能用之前缓存的数据。用户可能在不同时间点问同一问题，市场已经变了。

**T+1速查（再次强调）**：
| 买入日 | 解锁日 | 锁仓天数 |
|--------|--------|----------|
| 周一 | **周二** | 1天 |
| 周二 | **周三** | 1天 |
| 周三 | **周四** | 1天 |
| 周四 | **周五** | 1天 |
| 周五 | **下周一** | 3天（含周末） |

**无论RSI多低、布林带多贴下轨、回调多深，大盘弱时一律不推荐买入。**

2026-07-16：早盘创业板-2.48%、深证-1.91%，我仍然在分析RSI超卖、布林下轨、回调幅度，给出"东睦RSI 26.3可以买"、"芯片ETF target合理"的建议。用户反复问"挂单价是否合理"，我每次都回答"合理不用调"。结果收盘创业板-3.00%，三只全崩，当日亏470元。

用户原话：**"为啥今天我一直让你看挂单价 你一直觉得合理呢？现在亏损这么多 本来没必要亏损的啊 这种行情"**

**铁律**：大盘-2%控制仓位，大盘-3%暂停买入。这条写在第4步，但它是第0步——在拉任何个股数据之前就必须执行。

### 买入分析硬性门禁

回答"能买吗"/"挂单合理吗"之前，**必须先过这三关**：


**❌ 禁止**：大盘-2%时说"RSI超卖是🥇最强信号但大盘不好，建议小仓位"——这就是今天的问题。大盘不好就是不好，没有"但是"。

### 持仓股操作时的风控

大盘-2%以上时，持仓分析正常进行（止盈止损照常提醒），但：
- 不推荐加仓
- 不推荐新买入
- 只分析是否该减仓/止损

### 🔴 用户说做T时不要说割肉（2026-07-17教训）

**场景**：芯片ETF触发止损，我说"3.00出掉"。用户生气："我要做t你不理解吗？你现在让我割肉？"

**核心**：用户在大盘暴跌时的偏好是**做T降成本**，不是止损割肉。先给做T方案，止损是最后选项。

**✅ 正确流程**：
1. 用户问"怎么操作" → 先给做T方案（买入低位+反弹卖旧份额）
2. 计算做T降本效果：反弹+5%新成本多少
3. 止损只说一次，不反复催促

**❌ 禁止**：反复说"触发止损必须卖" → 用户要的是降本方案，不是逃命指令

## 🔴 基金 ≠ 股票：暴跌日不适用"等企稳"（2026-07-17教训）

**用户纠正**：大盘-4%时基金问能不能加仓，我反复说"等企稳"。用户质问："你总说等企稳，等稳定了还有抄底什么事？"

**核心区别**：

| | 个股 | 基金 |
|------|------|------|
| 归零风险 | 有（退市/暴雷） | **无**（一篮子几十只） |
| 单日极限跌幅 | -10%（跌停） | -6%基本是极限 |
| 反弹概率 | 不确定 | **极高**（急跌后必反弹） |
| "企稳"意义 | 必须等（防踩雷） | **不必等**（跌-6%就是买点） |

**✅ 正确**：基金单日-6%+大盘暴跌 → 可以建议加仓，给出具体额度和降本效果
**❌ 错误**：基金也用个股逻辑，反复说"等企稳" → 踏空反弹（7/9 +5.97%、7/14 +4.36%）

**规则**：
- 大盘-3%：**个股**暂停买入，**基金**可以建议加仓（说明理由+降本效果）
- 大盘-4%+：个股基金都暂停
- 周五买基金：提醒锁仓到下周一确认份额，但净值低时仍可考虑

**基金做T**：利用先进先出规则，低净值买入 + 反弹后赎回最早（成本最高）批次 → 总份额不变，成本下降。详见 `stock-trading-system` 的 `references/fund-t-trade.md`。

### 加仓判断标准（2026-07-06更新，纳入铁律）

**参照 `stock-trading-system` 5条铁律：**

1. 距止损 >5% 才考虑加仓（逼近止损的不补）— 铁律①
2. 成本涨幅 <10% 才考虑（快止盈的不追）— 铁律②
3. 当日涨幅 >5% 的不追高
4. 大盘跌超-3% → 暂停一切买入操作
5. 铁律④：已持2只则不加第3只
6. ETF止损用3%（铁律⑤），加仓后用加权成本重算止损
7. 观察股RSI 40-60、近5日回调>5%为最佳买点

### 💰 补仓成本计算公式
```python
# 加权平均成本
new_cost = (old_shares * old_cost + new_shares * new_price) / (old_shares + new_shares)
# 例: 原1600股@3.919，补500股@3.749
# → (1600*3.919 + 500*3.749) / 2100 = 3.879
```
补仓后止损线也需重算：`new_sl = new_cost * (1 - sl_pct/100)`

## 注意事项

1. **🔴 大盘风控第一**：任何买入分析前先查三大指数，任一指數≤-2%直接拒绝 → 详见 `references/2026-07-16-lesson.md`
2. **不要追高**：扫描结果如果今天涨停，明天高开不要追
3. **小仓位试探**：观察股票最多用1/6资金
4. **严格止损**：跌破止损位必须走
4. **结合持仓**：分析时要考虑已有持仓的风险
5. **⚠️ T+1风险分析（出错过！必须用速查表）**：

**T+1 速查表（按买入日查解锁日）：**

| 买入日 | 解锁日 | 锁仓天数 |
|--------|--------|----------|
| 周一 | **周二** | 1天 |
| 周二 | **周三** | 1天 |
| 周三 | **周四** | 1天 |
| 周四 | **周五** | 1天 |
| 周五 | **下周一** | 3天（含周末） |
| 节前最后交易日 | **节后首日** | N天 |

**⚠️ 教训（2026-06-29）**：周一买入贤丰控股，错误说成"周三可卖"，实际是**周二**。T+1就是下个交易日，不要脑补多算一天。

   - 区分可卖持仓 vs 锁仓持仓（今日买入的股票）
   - 周五买入锁仓2天（周六、周日），下周一才能操作
   - 止损建议只能针对可卖部分
   - 提醒用户锁仓风险，周末/假期前谨慎加仓
6. 盈利后更新本金：每次盈利卖出后，更新监控脚本的`TOTAL_CAPITAL`和`docs/system-logic.md`

## ⚠️ API不可用时的兜底策略（2026-07-14）

### 板块资金API — 早盘常返回rc=102

`fs=m:90+t2` 在刚开盘时经常返回 `rc=102`（无数据），不是bug，是东财数据延迟。10:00后通常恢复。

**策略**：
1. 先尝试获取，rc=102时标注"板块数据暂不可用"，不要反复重试
2. 用个股涨跌方向代替：如果芯片ETF-3%而大盘-0.5%，芯片板块大概率在流出
3. 盘中10:00后重试一次即可

### K线API (push2his) — 限流后返回空或curl 52

`push2his.eastmoney.com` 频繁返回空响应或 `RemoteDisconnected`。

**兜底链**（由快到慢）：
1. `curl -s push2his` → 空响应时**加User-Agent+Referer头**：
   ```bash
   curl -s -H "User-Agent: Mozilla/5.0" -H "Referer: https://quote.eastmoney.com/" "URL"
   ```
   ⚠️ 这个header组合在7/21验证有效，空响应时优先用这个。
2. `stock-screener get_kline_data()` → 有代理池+Cookie，但可能返回None
3. **手工计算RSI**：用已知收盘价列表 + Python 直接计算，不需要完整K线API。详见 `references/manual-rsi-calculation.md`

### 新闻搜索 — eastmoney So不可靠

`so.eastmoney.com/news/s` 返回空在curl下常见（反爬）。

**替代**：直接用个股涨跌+板块表现推断，标注"新闻获取失败"。不要因为新闻搜不到就跳过整个分析流程。

## ⚠️ 回本挂单股数陷阱（2026-07-13）

**东睦案例**：亏损854，挂30.58买200股 → 回本需涨到34.85(+14%)，几乎不可能。
```python
回本卖出价 = 卖出均价 + |亏损| / 股数
# 200股: 33.22 + 854/200 = 37.49 — 不可能
# 300股: 33.22 + 854/300 = 36.07 — 到止盈位就够了
# 400股: 33.22 + 854/400 = 35.36 — 更容易
```
**规则**：设挂单前先算 `股数 ≥ |亏损| / (止盈价 - 挂单价)`，确保止盈位能覆盖回本。

## 🔔 做T推送规范（2026-07-17教训）

**用户两次纠正推送问题**：
- "推送啊你倒是。。" → 检测到T目标到达后立刻推，不要等用户问
- "是让我多少卖啊 你也没写清楚啊" → 推送必须写明**卖多少股**、**卖什么价**

### ✅ 正确推送格式

```
🎯 【XX股票 做T卖出指令】

卖出 300股 @ 28.55
（补300@27.19  T赚+408）

⚠️ 卖的是旧持仓，不是今天买的
```

必须包含：卖出数量、卖出价格、补仓价、T赚金额、区分锁仓/可卖

### ❌ 禁止

- 模糊表述（"卖旧持仓中对应数量"）→ 必须写具体数字
- 推到企微后等用户反应 → 价格到了 = 立刻推
- 只说T赚不说卖多少 → 用户无法操作

### T目标价更新后的推送流程

1. 更新 `state['t_targets']` → 降低到实际可触发的价位（+2%而不是+5%）
2. 如果现价已超过新目标 → 立即用 `send_wx` 直接推
3. 同时写 `pending_trades` 备份，确保cronjob也能推
4. 不要假设"cronjob会自己推" → 主动推一次
