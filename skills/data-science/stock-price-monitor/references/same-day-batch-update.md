# 同日多笔交易批量更新

> 场景：用户在一次消息中报告多笔交易，或同一天内连续多次操作同一股票。
> 例：7/21 — "东睦清了25.8。新疆卖了800，9.04。400股9.05"

## 批量更新策略

### 同一股票同一天：减仓 → 清仓

当同一股票先减仓再清仓（如东睦：减500@25.50 → 清500@25.80），全天的净效果就是清仓。

**处理方式**：
1. **不维护中间状态**：第二次更新直接覆盖第一次，最终该股票 `cost=None, shares=0`
2. **累计亏损合并计算**：
   ```
   减仓亏损 = 减仓股数 × (减仓价 - 成本)
   清仓亏损 = 清仓股数 × (清仓价 - 成本)
   总亏损 = 减仓亏损 + 清仓亏损
   ```
3. **TOTAL_CAPITAL 只更新一次**：用总亏损一次性调整
4. **pending_trades 可以追加**：减仓通知 + 清仓通知都入队

### 不同股票同一天：批量清仓

当用户一次消息报告多只股票清仓（如东睦+新疆众和）：

**处理方式**：
1. **一个 commit 覆盖全部**：`交易: 东睦清仓(-X) + 众和清仓(-Y)`
2. **TOTAL_CAPITAL 一次性调整**：总亏损 = 各股亏损之和
3. **pending_trades 多条入队**：每只股票一条通知
4. **状态清理批量执行**：循环清理每只股票的 alerted/prev_limit_up/prev_limit_down

### 验证要点

```python
# 批量更新后验证
new_total = old_total_capital - sum(各股亏损)
# 核对：new_total = 现金 + 剩余持仓市值
```

## 效率优化

相比逐笔更新的6步流程，批量更新节省：
- 减少 Git commit 次数（N笔→1次）
- 减少语法验证次数
- TOTAL_CAPITAL 只算一次

但注意：
- 每只股票的注释和配置块仍需分别更新
- LOSS_HISTORY 每只股票独立维护
- pending_trades 可以为每笔交易各写一条
